### Test coverage

### Lessons learned

1. **Elixir Bitwise operators require explicit import**: `<<<`, `>>>`, `|||`, `&&&` are not available by default. Must `import Bitwise` in any module using them, including test files.

2. **Ecto schema type `:text` does not exist**: In migrations you use `:text` for the DB column type, but in Ecto schemas you must use `:string` for both varchar and text columns. `:text` is only valid in migrations.

3. **`:utc_datetime_usec` requires microsecond precision (6 decimal places)**: `DateTime.from_unix!(ms, :millisecond)` creates a DateTime with precision=3 (milliseconds), e.g. `~U[2026-02-21 23:23:22.291Z]`. Ecto's `:utc_datetime_usec` rejects this. Fix: use `DateTime.from_unix!(ms * 1000, :microsecond)` to get precision=6. Note: `DateTime.truncate(:microsecond)` does NOT upgrade precision — it only truncates downward and preserves the original precision marker.

4. **`replace_all` in edits catches alias lines too**: When doing a global replace of `Slackex.Repo` -> `Repo`, the `alias Slackex.Repo` line also gets replaced to `alias Repo` which is invalid. Always be careful with global replacements near alias declarations.

5. **Phoenix 1.8 no longer includes Swoosh by default but still generates mailer code**: After `mix phx.new`, manually remove `lib/slackex/mailer.ex` and all Swoosh config if email is not needed.

6. **Credo alias ordering**: Credo --strict checks that aliases within a group are alphabetically ordered. Multi-alias `{A, B, C}` must also be sorted. `Slackex.Accounts` should come before `Slackex.Accounts.{Auth, Guardian}` which comes before `Slackex.Repo`.

7. **Credo nested module aliases**: When a module is used more than once, Credo prefers a top-level `alias` rather than inline fully-qualified names. Use `# credo:disable-for-this-file Credo.Check.Design.AliasUsage` for Phoenix-generated files.

8. **Credo function nesting depth**: `with` clauses containing nested `case`/`if` easily exceed max depth of 2. Extract the inner logic into private functions to flatten the nesting.

## Phase 1 Foundation — API & WebSocket Layer (2026-02-21)

### What was built
- Shared JSON serializers: UserJSON, ChannelJSON, MessageJSON (Snowflake IDs as strings for JS safety)
- API auth pipeline: Guardian plug pipeline (VerifyHeader, EnsureAuthenticated, LoadResource) with JSON 401 error handler
- API auth controller: POST /api/auth/login (email+password -> tokens+user), POST /api/auth/refresh (token rotation)
- API bootstrap controller: GET /api/bootstrap (JWT-protected, returns user+channels+DMs+unread_counts)
- UserSocket: JWT-authenticated WebSocket with chat:* and dm:* channel routes
- ChatChannel: join (role check, load messages, mark as read), send message (broadcast via Phoenix Channel)
- DMChannel: join (participant check, load messages), send message (broadcast)
- ChannelCase test support module for Phoenix Channel tests
- Chat context additions: get_dm/1, list_dm_messages/2

### Test coverage
- 105 tests (0 failures), mix credo --strict (0 issues), mix compile --warnings-as-errors clean
- 39 new behavioral tests covering: API login, refresh, bootstrap, WebSocket connect/join/send, serializer correctness

### Bugs found and fixed during testing

1. **Guardian.Plug.Pipeline requires `otp_app`**: `use Guardian.Plug.Pipeline` without `otp_app: :slackex` causes a RuntimeError at runtime. Not caught by compiler — only surfaced when tests actually exercised the pipeline.

2. **Phoenix JSON view namespace must match controller namespace**: Controllers in `SlackexWeb.API.*` expect JSON views in the same namespace. Defining serializers as `SlackexWeb.MessageJSON` instead of `SlackexWeb.API.MessageJSON` causes Phoenix to fail to find the view module. The convention is: controller `SlackexWeb.API.AuthController` renders via `SlackexWeb.API.AuthJSON`.

3. **`to_string/1` on non-atom Guardian error tuples**: `Auth.refresh_api_token/1` can return `{:error, %ErlangError{}}` for malformed tokens. Calling `to_string(reason)` on a struct raises `Protocol.UndefinedError`. Guard clause needed to handle non-atom/binary error reasons.

4. **Nil sender crashes serializer**: `MessageJSON.data/1` called `UserJSON.data(sender)` without nil guard. When `sender` is nil (deleted user), `UserJSON.data/1` pattern match fails with `FunctionClauseError`. Fixed with a `serialize_sender/1` private function that returns nil for nil input.

### Lessons learned

9. **Phoenix JSON view naming convention**: For controllers namespaced under `SlackexWeb.API.*`, the JSON view must also be under `SlackexWeb.API.*`. Phoenix resolves view modules by replacing `Controller` with `JSON` in the controller module name.

10. **Guardian.Plug.Pipeline needs `otp_app`**: Always pass `otp_app: :your_app` to `use Guardian.Plug.Pipeline`. Without it, Guardian cannot find the configuration and raises at runtime, not compile time.

11. **Serializers need nil guards for optional associations**: Any serializer field that calls another serializer must guard against nil. Use a private helper: `defp serialize_foo(nil), do: nil; defp serialize_foo(foo), do: FooJSON.data(foo)`.

12. **Snowflake GenServer is auto-started in test**: `Slackex.Infrastructure.Snowflake` is in the application supervision tree. Do NOT call `start_supervised!` in test setup — it will raise `{:already_started, pid}`. Only start it manually if the application supervisor is not running.
