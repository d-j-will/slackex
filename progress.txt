### Lessons learned

1. **Elixir Bitwise operators require explicit import**: `<<<`, `>>>`, `|||`, `&&&` are not available by default. Must `import Bitwise` in any module using them, including test files.

2. **Ecto schema type `:text` does not exist**: In migrations you use `:text` for the DB column type, but in Ecto schemas you must use `:string` for both varchar and text columns. `:text` is only valid in migrations.

3. **`:utc_datetime_usec` requires microsecond precision (6 decimal places)**: `DateTime.from_unix!(ms, :millisecond)` creates a DateTime with precision=3 (milliseconds), e.g. `~U[2026-02-21 23:23:22.291Z]`. Ecto's `:utc_datetime_usec` rejects this. Fix: use `DateTime.from_unix!(ms * 1000, :microsecond)` to get precision=6. Note: `DateTime.truncate(:microsecond)` does NOT upgrade precision — it only truncates downward and preserves the original precision marker.

4. **`replace_all` in edits catches alias lines too**: When doing a global replace of `Slackex.Repo` -> `Repo`, the `alias Slackex.Repo` line also gets replaced to `alias Repo` which is invalid. Always be careful with global replacements near alias declarations.

5. **Phoenix 1.8 no longer includes Swoosh by default but still generates mailer code**: After `mix phx.new`, manually remove `lib/slackex/mailer.ex` and all Swoosh config if email is not needed.

6. **Credo alias ordering**: Credo --strict checks that aliases within a group are alphabetically ordered. Multi-alias `{A, B, C}` must also be sorted. `Slackex.Accounts` should come before `Slackex.Accounts.{Auth, Guardian}` which comes before `Slackex.Repo`.

7. **Credo nested module aliases**: When a module is used more than once, Credo prefers a top-level `alias` rather than inline fully-qualified names. Use `# credo:disable-for-this-file Credo.Check.Design.AliasUsage` for Phoenix-generated files.

8. **Credo function nesting depth**: `with` clauses containing nested `case`/`if` easily exceed max depth of 2. Extract the inner logic into private functions to flatten the nesting.

### Bugs found and fixed during testing

1. **Guardian.Plug.Pipeline requires `otp_app`**: `use Guardian.Plug.Pipeline` without `otp_app: :slackex` causes a RuntimeError at runtime. Not caught by compiler — only surfaced when tests actually exercised the pipeline.

2. **Phoenix JSON view namespace must match controller namespace**: Controllers in `SlackexWeb.API.*` expect JSON views in the same namespace. Defining serializers as `SlackexWeb.MessageJSON` instead of `SlackexWeb.API.MessageJSON` causes Phoenix to fail to find the view module. The convention is: controller `SlackexWeb.API.AuthController` renders via `SlackexWeb.API.AuthJSON`.

3. **`to_string/1` on non-atom Guardian error tuples**: `Auth.refresh_api_token/1` can return `{:error, %ErlangError{}}` for malformed tokens. Calling `to_string(reason)` on a struct raises `Protocol.UndefinedError`. Guard clause needed to handle non-atom/binary error reasons.

4. **Nil sender crashes serializer**: `MessageJSON.data/1` called `UserJSON.data(sender)` without nil guard. When `sender` is nil (deleted user), `UserJSON.data/1` pattern match fails with `FunctionClauseError`. Fixed with a `serialize_sender/1` private function that returns nil for nil input.

### Lessons learned

9. **Phoenix JSON view naming convention**: For controllers namespaced under `SlackexWeb.API.*`, the JSON view must also be under `SlackexWeb.API.*`. Phoenix resolves view modules by replacing `Controller` with `JSON` in the controller module name.

10. **Guardian.Plug.Pipeline needs `otp_app`**: Always pass `otp_app: :your_app` to `use Guardian.Plug.Pipeline`. Without it, Guardian cannot find the configuration and raises at runtime, not compile time.

11. **Serializers need nil guards for optional associations**: Any serializer field that calls another serializer must guard against nil. Use a private helper: `defp serialize_foo(nil), do: nil; defp serialize_foo(foo), do: FooJSON.data(foo)`.

12. **Snowflake GenServer is auto-started in test**: `Slackex.Infrastructure.Snowflake` is in the application supervision tree. Do NOT call `start_supervised!` in test setup — it will raise `{:already_started, pid}`. Only start it manually if the application supervisor is not running.

### Test coverage
- 157 tests (0 failures), mix credo --strict (0 issues), mix compile --warnings-as-errors clean

### Lessons learned

13. **ETS named tables are owned by the creating process**: When `Cache.Local` is added to the application supervisor, `start_supervised!(Local)` in tests raises `{:already_started, pid}`. Fix: use `:ets.delete_all_objects(:slackex_message_cache)` in test `setup` to clear state between tests rather than restarting the GenServer.

14. **Task.Supervisor start_supervised! conflict**: Once `{Task.Supervisor, name: Slackex.WriteSupervisor}` is in the app supervisor, test setups that call `start_supervised!({Task.Supervisor, name: Slackex.WriteSupervisor})` fail with `:already_started`. Remove the per-test `start_supervised!` and rely on the running supervisor.

Bounded-queue tests should either use many distinct users or be removed in favour of testing the limit via direct ETS/queue inspection.

### Test coverage
- 183 tests (0 failures), mix credo --strict (0 issues), mix compile --warnings-as-errors clean
- 26 new behavioral tests covering: Presence tracking, typing broadcast, HistoryLoader cache hit/miss/before pagination, CacheWarmer perform/active channels query, ChatChannel/DMChannel write path migration

### Lessons learned

17. **Duplicate broadcast trap when migrating write paths**: When migrating from `Chat.send_message` (direct DB write) to `Messaging.send_message` (ChannelServer), you must remove the channel's `broadcast!/3` call. The ChannelServer already broadcasts via PubSub — keeping both causes duplicate messages. Instead, subscribe the channel process to the PubSub topic on join and handle `{:new_message, msg}` to push to the client.

18. **HistoryLoader returns different shapes than Chat context**: `Chat.list_messages/2` returns Ecto structs with preloaded `:sender`. `HistoryLoader.recent/2` returns cached maps (no `:sender`). For the initial channel join payload (which needs full serialization via MessageJSON.data/1), keep using `Chat.list_messages` directly. Use HistoryLoader for the CQRS read path where map shapes suffice.

19. **Oban test config `testing: :inline`**: This makes all Oban jobs execute synchronously in tests. Without it, jobs are enqueued but never run, causing test assertions on side effects to fail.

20. **struct_to_map for cache backfill**: When backfilling ETS cache from DB results, Ecto structs must be converted to maps via `Map.from_struct/1`. Direct ETS storage of structs works but is inconsistent with the map-based cache contract used by ChannelServer.

## Phase 2 CQRS — Steps 8–9: Envelope Contract & Write Rejection Normalization (2026-02-22)

### What was built
- **`Slackex.Messaging.Envelope`** (`lib/slackex/messaging/envelope.ex`): Versioned envelope module wrapping all realtime payloads. `wrap/4` converts `{:channel, id}` or `{:dm, id}` target tuples into `%{v: 1, event, target: %{type, id}, payload, meta: %{sent_at, correlation_id}}`. `unwrap/1` extracts event, target, payload for pattern matching.
- **ChannelServer updates** (`lib/slackex/messaging/channel_server.ex`): Broadcasts now use `{:envelope, envelope}` instead of raw `{:new_message, message}`. Added `check_backpressure/1` — returns `{:error, :backpressure}` when `pending_writes >= @max_pending_writes` (was silently dropping). Added `validate_content/1` — returns `{:error, :invalid_content}` for empty or >4000 char content. Both checks run before Snowflake ID generation in the `with` chain.
- **Messaging context** (`lib/slackex/messaging/messaging.ex`): `broadcast_typing/2` now wraps typing events in envelope.
- **ChatChannel + DMChannel** (`lib/slackex_web/channels/`): Handle `{:envelope, %{event: "message.new"}}` and `{:envelope, %{event: "typing"}}` patterns. Push events as `"message.new"` and `"typing"` (not `"new_message"`). All write rejection errors normalized to `%{reason: string, message: string}` shape with specific clauses for `:rate_limited`, `:backpressure`, `:unauthorized`, `:invalid_content`.
- **DMChannel PubSub bug fix**: Added missing `Phoenix.PubSub.subscribe(Slackex.PubSub, "dm:#{dm_id}")` in `join/3` — DM real-time message delivery from ChannelServer was broken.
- **Contract tests** (`test/slackex_web/channels/envelope_contract_test.exs`): 12 contract tests tagged `@tag :contract` covering write rejection shapes and envelope delivery for both ChatChannel and DMChannel.

### Test coverage
- 206 tests (0 failures), mix credo --strict (0 issues), mix compile --warnings-as-errors clean
- 23 new tests: 6 envelope unit, 5 ChannelServer validation, 12 contract tests
- Contract tests excluded from default run; `mix test --include contract` runs all 206

### Lessons learned

21. **Silent drops become invisible bugs**: ChannelServer was silently dropping messages when `pending_writes` was full — the caller got `{:ok, message}` even though the message would never be persisted. The spec correctly required `{:error, :backpressure}` so the client knows the message was rejected. Always fail visibly at system boundaries.

22. **DMChannel PubSub subscription was missing**: DMChannel.join did not call `Phoenix.PubSub.subscribe/2`, so `handle_info({:envelope, _})` would never fire for DM messages delivered by ChannelServer. ChatChannel had the subscription but DMChannel didn't — copy-paste divergence. The bug was masked in tests because Phoenix channel test helpers may auto-subscribe. Always verify PubSub wiring with integration-level assertions.

23. **Contract tests as executable protocol spec**: Tagging tests with `@tag :contract` and excluding from default run (`ExUnit.configure(exclude: [:contract])`) creates a clear separation between implementation tests and protocol-level contracts. `mix test` runs fast behavioral tests; `mix test --include contract` adds protocol verification. This pattern from the spec (Section 6.4) avoids coupling fast CI feedback with slower contract checks.

24. **Content validation belongs in the GenServer, not the schema**: ChannelServer bypasses Ecto changesets (builds maps directly for performance). This means Message schema's 1-4000 char validation never runs. Content validation must be duplicated in the GenServer's write path. The check must run before Snowflake ID generation to avoid wasting IDs on invalid messages.

### Lessons learned

25. **Login forms must POST to a controller, not LiveView**: LiveView runs over WebSocket. Session cookies can only be set during an HTTP request/response cycle. Login forms use `action={~p"/users/log-in"}` with `phx-update="ignore"` to submit as a standard form POST to UserSessionController, which calls `Plug.Conn.put_session/3`. Registration can stay in LiveView (redirect to login page on success).

26. **Layout template embedding conflict**: Phoenix's `embed_templates` macro embeds all `.heex` files matching the pattern. If you define both an inline function component `def app(assigns)` AND a template file `layouts/app.html.heex`, the inline function shadows the template. Use `embed_templates "layouts/*"` to embed all layout templates and remove inline definitions.

27. **LiveView streams for efficient message lists**: Using `stream(:messages, messages)` in mount and `stream_insert(:messages, enriched_message)` in PubSub handlers gives Phoenix's diffing engine efficient DOM updates. Each message needs a unique DOM ID (`:dom_id` option or `id` field).

28. **PubSub message enrichment**: ChannelServer broadcasts message maps without preloaded associations. The LiveView must enrich incoming PubSub messages by fetching the sender user: `Accounts.get_user!(message.sender_id)`. This is acceptable since PubSub messages arrive one at a time (not in bulk).

29. **ConnCase test helpers for LiveView auth**: `log_in_user/2` generates a session token via `Accounts.generate_user_session_token/1` and puts it in the test session via `init_test_session/2` + `put_session(:user_token, token)`. This mirrors what UserAuth does in production without needing HTTP roundtrips.

30. **Phoenix 1.8 colocated hooks break esbuild if unused**: Phoenix 1.8 scaffolds `app.js` with `import {hooks as colocatedHooks} from "phoenix-colocated/slackex"`. If no colocated hooks exist, this module is never generated and esbuild fails silently in dev (no error in terminal). LiveSocket never connects, so LiveView forms fall back to standard HTTP POST to the current URL, producing confusing "no route found" errors. Fix: remove the import and the spread from the hooks object if not using colocated hooks. **Critical**: `mix test` does NOT catch this — LiveView tests bypass JavaScript entirely. Always run `mix assets.build` after LiveView/JS changes to verify the frontend compiles.

### Lessons learned

31. **Horde is a drop-in replacement for stdlib Registry/DynamicSupervisor**: The API is nearly identical. `{:via, Registry, {Name, key}}` becomes `{:via, Horde.Registry, {Name, key}}`. `DynamicSupervisor.start_child` becomes `Horde.DynamicSupervisor.start_child`. The main difference is configuration: `members: :auto` for automatic cluster discovery and `process_redistribution: :active` for rebalancing.

32. **Test files reference supervisor/registry APIs directly**: When migrating from stdlib to Horde, test files that call `DynamicSupervisor.terminate_child`, `DynamicSupervisor.which_children`, or `Registry.lookup` must also be updated. These are not caught by `mix compile` — they only surface when running `mix test`.

33. **`replace_all` can double-prefix existing Horde references**: If a file already has some `Horde.DynamicSupervisor` references and you do `replace_all` of `DynamicSupervisor` → `Horde.DynamicSupervisor`, the existing ones become `Horde.Horde.DynamicSupervisor`. Always check for pre-existing prefixed references before bulk replacement, or do a targeted replace instead.

34. **Horde.DynamicSupervisor child_spec needs explicit members**: Unlike stdlib DynamicSupervisor which just needs `name:`, Horde requires `members: :auto` (or explicit member list) in the child spec. Without it, the supervisor starts but doesn't participate in the distributed cluster.

## Phase 3 Distribution — Step 2.2: Writer Fencing (2026-02-22)

### What was built
- **`Slackex.Pipeline.BatchWriter`** (`lib/slackex/pipeline/batch_writer.ex`): Replaced `insert_batch/1` with epoch-fenced `insert_batch/2(messages, opts)`. Required keyword opts: `epoch:` (caller's writer epoch), `type:` (`:channel` or `:dm`), `id:` (channel/DM integer ID). Insert is wrapped in `Repo.transaction` with `SELECT writer_epoch FROM {table} WHERE id = $1 FOR UPDATE` row lock. If `db_epoch > caller_epoch`, transaction rolls back with `{:error, :epoch_stale}`. Updated `async_insert_batch` to 3-arg signature `async_insert_batch(messages, caller_ref, opts)` passing epoch opts through. Old `/1` arities removed — all callers must provide epoch.
- **`Slackex.Messaging.ChannelServer`** (`lib/slackex/messaging/channel_server.ex`): Acquires writer epoch atomically in `init/1` via `UPDATE {table} SET writer_epoch = writer_epoch + 1 WHERE id = $1 RETURNING writer_epoch`. Epoch stored in GenServer state. All three BatchWriter call sites (batch_flush, retry, idle timeout) updated to pass `epoch_opts(state)`. New `handle_info({:batch_result, _ref, {:error, :epoch_stale}}, state)` handler: emits `[:slackex, :channel_server, :epoch_stale_shutdown]` telemetry, drops all pending/in-flight writes, terminates with `{:stop, :normal, state}`. Stale guard in `send_message` returns `{:error, :not_writer}` when `state.stale == true`.

### Lessons learned

35. **`FOR UPDATE` row lock inside Repo.transaction eliminates TOCTOU races**: The epoch check and batch insert must happen atomically. Without `FOR UPDATE`, a concurrent writer could increment the epoch between the SELECT and INSERT, allowing a stale writer to persist messages. The row lock serializes access — only one transaction can hold the lock at a time.

36. **Epoch_stale tests must use dedicated channels**: Tests that trigger `{:stop, :normal, state}` on a ChannelServer cause Horde to attempt a restart. The restarted process tries to acquire an epoch (DB call) using the dying test's sandbox connection, which crashes. Fix: create a fresh channel per destructive test instead of using the shared setup channel. This isolates the blast radius of process termination.

37. **Credo catches `Ecto.Adapters.SQL.query!` as a nested module call**: When using raw SQL via `Ecto.Adapters.SQL.query!/3`, Credo's alias-usage check flags it. Add `alias Ecto.Adapters.SQL` at the top of the module and use `SQL.query!/3` instead.

38. **Writer epoch as a fencing token is a classic distributed systems pattern**: The epoch acts as a monotonic "fencing token" — each new writer gets a strictly higher epoch, and the database rejects writes from stale epochs. This makes the single-writer invariant verifiable rather than merely assumed. Unlike distributed locks (which can expire), the epoch persists in the database and is checked at write time.
